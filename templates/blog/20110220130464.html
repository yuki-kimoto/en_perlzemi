perlbrew - Install Perl in your environment

To install Perl in your environment, use a tool called perlbrew. This is useful if you don't want to use the system standard Perl or if you want to use a newer version of Perl. You can also use perlbrew to switch between multiple versions of Perl.

Install perlbrew

Execute the following command. 
curl -L http://install.perlbrew.pl | bash
If you don't have the curl command, try running the following command: 
# Linux
wget -O - http://install.perlbrew.pl | bash

# FreeBSD
fetch -o- http://install.perlbrew.pl | sh
You can also install from cpan in two steps. 
sudo cpan App::perlbrew
perlbrew init
Next, let's pass the path of perlbrew. Add the configuration file for bash provided by perlbrew to "~ / .bashrc" (in the case of bash). 
echo "source ~ / perl5 / perlbrew / etc / bashrc" >> ~ / .bashrc
Reload ".bashrc". 
source ~ / .bashrc

Installing Perl in the user environment

Install Perl in your environment. Let's install Perl for 5.28.2. It will take some time, so wait (about 30 minutes). 
perlbrew install perl-5.28.2
If you can't install it, the tools needed to compile Perl may not have been installed yet. The following tools are required, so install them according to each environment.
  • make
  • gcc
  • patch
If you want to force the test to skip and force the installation, use the "-n" option. 
perlbrew install -n perl-5.28.2
Let's take a look at the list of available Perls. Use the perlbrew list command. 
perlbrew list
The available Perls are displayed as shown below. The one with the asterisk is the Perl you are currently using. 
  perl-5.28.2
* / usr / bin / perl (5.8.8)
Use perlbrew's switch command to switch Perl. 
perlbrew switch perl-5.28.2
Let's actually check the version of Perl. 
perl -v
The version is 5.28.2 as shown below. 
This is perl 5, version 16, subversion 3 (v5.28.2) built for i686-linux

Install cpanm

Let's also install cpanm, which is convenient for installing modules. 
perlbrew install-cpanm
Now you can install the module with the following command. 
cpanm JSON
The module will be installed in the following directories. 
/home/kimoto/perl5/perlbrew/perls/perl-5.28.2/lib/site_perl/5.28.2/

Revert to original Perl

To revert to the original Perl: 
perlbrew off

How to use the version of Perl specified by perlbrew with crontab

One of the main points of perlbrew is that the version of Perl specified by perlbrew is not used by cron. This is because .bash_profile is not loaded when running from cron because the shell is not started. The setting by perlbrew is described in .bash_profile. The simplest solution is to rewrite PATH in crontab to the value of the environment variable PATH that the user has. First, as the user who installed perlbrew, display the value of the PATH environment variable. 
env | grep PATH
You will see some environment variables that include PATH, so look for something called PATH. (This is an example when the user name is admins.) 
PATH = / home / admins / perl5 / perlbrew / bin: / home / admins / perl5 / perlbrew / perls / perl-5.16.2 / bin: / usr / kerberos / bin: / usr / local / bin: / bin: / usr / bin: / home / admins / bin
Edit crontab with the following command. 
crontab -e
Then write the above PATH value to the top of crontab. 
PATH = / home / admins / perl5 / perlbrew / bin: / home / admins / perl5 / perlbrew / perls / perl-5.16.2 / bin: / usr / kerberos / bin: / usr / local / bin: / bin: / usr / bin: / home / admins / bin

# cron settings continue ...
Now cron will use the specified version of Perl.

First line description of Perl script

Write the first line of the Perl script using the env command as shown below. 
# ! / usr / bin / env perl
This will automatically detect perl from the set PATH value. As you can see, this method solidifies the PATH, so crontab is not portable at all. Since the user name admins is directly embedded and the version number of perl is written as it is, it will not work when changing the version with perlbrew. But I think this method is the best. The reason is that it is difficult to make it portable because there is a limitation that environment variables cannot be expanded in crontab. Portability increases the amount of description, requires additional scripts, and complicates management. Also, no matter how hard you try, crontab usually has some differences depending on the environment, so even if you keep portability, there may not be much merit.
